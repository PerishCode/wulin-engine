# Experiment 0082: Plain Prototype v0 Stage Seal

Status: Accepted

## Hypothesis

The accepted runtime, host, actor, and operator boundaries already form one coherent plain game-
engine Prototype v0: a source-free cold start can open the sole canonical renderer, drive one
grounded animated actor from native input through fixed simulation, follow it with one camera, and
enter engine-owned composition traversal without a diagnostic application mode. This boundary can
be sealed by existing maintained workflows without adding product behavior or concealing its
finite-world limits.

## Scope

Validate and name the current finite, single-actor prototype stage. Run the maintained product
operator from absent generated prototype sources/config, the focused real-process prototype gate,
and one long canonical-runtime acceptance checkpoint.

Update current/public documentation only after all three layers pass. Add no runtime, application,
wrapper, support, manifest, format, shader, asset, or Wulin behavior. Do not introduce a version
switch, compatibility alias, inspect endpoint, periodic product telemetry, source-service claim, or
new feature to make the milestone appear broader.

Prototype v0 includes strict canonical bootstrap, a native window/input/time loop, one grounded
capacity-one actor, fixed gravity and W/A/S/D movement, transactional Survey/Walk presentation,
exact eight-way facing, actor-local clip time, one actor-relative camera, and one-time traversal
activation. Sustained product traversal evidence, an application-owned finite-world edge policy,
prefetch policy, horizontal velocity/acceleration, Run/blending/root motion, gameplay interaction,
multiple actors, networking, and Wulin content remain outside the stage.

## Workload

1. From a stopped process tree and absent generated prototype sources/config, run
   `runseal :prototype start`. Require deterministic 289-center / 441-region preparation, strict
   canonical readiness, and one live non-diagnostic prototype process.
2. Query status, restart without recooking, require replacement process identity, then stop and
   require zero owned target/broker processes.
3. Run `runseal :canonical-prototype`. Require strict startup failure, restart equality, native-W
   input, grounded fixed simulation, transactional presentation/facing/epoch, actor-relative camera,
   first traversal schedule, render-block disposition, and cleanup evidence through the live app.
4. Run `runseal :canonical-runtime` once as the stage checkpoint. Require the complete accepted
   renderer/content/traversal/resource/lifecycle suite to remain exact.
5. Run the merge-checkpoint guard and require a documentation-only product diff with no generated
   output or compatibility surface.

## Controlled Variables

- Product and canonical wrappers, support modules, Sidecar manifests, executable arguments, and
  readiness contracts are unchanged.
- The prototype sandbox remains zero-origin `[-8,8]²`, active radius 2, with prefetch disabled.
- Runtime, actor, input, clock, simulation, camera, traversal, renderer, GPU resources,
  synchronization, cookers, formats, assets, and Wulin ownership are unchanged.
- Existing focused gates retain their accepted tolerances and exact revision contracts. The stage
  seal does not duplicate them in a new aggregate wrapper.
- The long canonical workflow is run because this is a stage boundary, not as an edit-loop gate.

## Metrics

- Product cook counts, source/config bytes and hashes, preparation duration, readiness, process
  identity replacement, and final owned process count.
- Focused prototype test counts and exact actor motion/presentation/epoch, camera, traversal,
  failure, restart, and lifecycle evidence.
- Canonical runtime frame/capture/oracle hashes, traversal outcomes, same-process resource plateau,
  lifecycle cycles, duration, and final process state.
- Git diff ownership, Flavor denies/warnings, forbidden compatibility scan, and generated files.

## Acceptance Criteria

- A source-free `runseal :prototype start` alone prepares the documented deterministic sandbox and
  reaches canonical readiness; restart replaces the process without recooking and stop leaves no
  owned process.
- `runseal :canonical-prototype` passes the complete accepted live single-actor/input/simulation/
  camera/traversal boundary with no inspect endpoint or product-only observation path.
- `runseal :canonical-runtime` passes its complete renderer, source, traversal, resource, and
  lifecycle acceptance once at this stage boundary.
- The committed change contains only the experiment, ADR, current/public stage documentation, and
  required document registration/index metadata. There is no product-code, operator behavior,
  acceptance-support, manifest, source-format, GPU, asset, or Wulin delta and no generated output.
- `runseal :guard` passes. Documentation names only the bounded Prototype v0 claims above and keeps
  the finite sandbox, sustained product traversal, gameplay, multi-actor, and source-service gaps
  explicit.

## Reference Environment

The experiment uses the repository-pinned Windows/Rust/Deno toolchains, reference D3D12 GPU,
existing maintained Runseal operators, Sidecar process ownership, deterministic cookers, and sole
canonical runtime.

## Evidence

- From absent `out/cooked/prototype/` and bootstrap config, `runseal :prototype start` passed in
  13.3 seconds. The maintained wrapper therefore completed its exact 289 scheduled-center / 441
  source-region checks before Sidecar readiness. Generated terrain/object/config sizes were
  1,835,008 / 18,092,032 / 240 bytes with SHA-256
  `b93391dd...0b97` / `ca1f2163...5186` / `49353733...c3fe`.
- Initial product target PIDs were `5852/23580/21116`; restart replaced them with
  `33552/24988/26068` while all three hashes remained exact. Final stop reported empty target and
  broker PID sets.
- `canonical-prototype-v8` passed in 36.748 seconds with 77 engine-runtime, ten prototype, and 21
  reference-host tests. Existing strict startup failure, restart, native-W movement,
  Survey/Walk/facing/epoch, actor/camera/traversal, render-block, and zero-process cleanup evidence
  passed unchanged.
- The stage-only `canonical-runtime-v1` checkpoint passed in 744.8 seconds. Its 6,479,386-byte
  acceptance report recorded 32 reactive and 32 prepared traversal samples, an eight-sample
  64-publication resource sweep, active handle limit 532 / peak 531 / final quiescent 516, and 16
  complete lifecycle cycles. The report SHA-256 was `b2715687...b9b6`.
- The repository diff adds no runtime/application/support/manifest/format/shader/asset/Wulin change.
  Final registration, documentation checks, and the single merge-checkpoint guard passed with zero
  Flavor denies and five pre-existing warnings.

## Conclusion

Accepted. The current self-contained finite single-actor application is the plain Prototype v0
stage boundary. It is a real canonical engine loop, not an infinite-world, sustained traversal,
gameplay, multi-actor, or Wulin claim. No feature or diagnostic surface was added to create the
milestone.
